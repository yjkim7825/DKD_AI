# 논문(저자) vs 우리 재현 — 차이 정리

단계별로 묶음. 각 항목: **저자 방식 → 우리 방식 → 왜 → 차이**
(발표 때 "왜 다르냐" 대비용)

---

## 1. GEO 데이터

> 📌 전처리 코드 차이(저자 vs 우리)는 **`01_전처리_코드차이.md`** 에도 정리됨. 여기선 재현 차이 관점으로 유지.

### 1-1. probe → gene 매핑 방식
- **저자**: **주석 테이블(GPL/Brainarray CDF) 다운받아** 매핑
    - 104948·104954 = Brainarray 커스텀 CDF(폴더에 저자가 남김)
    - 원본 템플릿(data preprocessing 1.R) = series_matrix + GPL.txt
- **우리**: **Bioconductor 라이브러리**로 매핑 (다운 불필요)
    - 30529 → `hgu133a2.db` / 96804 → `hta20transcriptcluster.db`(oligo)
    - 104948·104954 → 저자 Brainarray CDF 그대로 따름
- **왜**: GPL.txt가 데이터별로 없거나 칩이 달라서, 표준 라이브러리(db)가 매핑표를 내장 → 다운 없이 처리. 저자 자료 있는 것만 그걸 따름
- **차이**: 방향 동일. 매핑 db에 따라 **유전자 수가 달라질 수 있음**(아래 1-3)

### 1-2. 정규화 / ComBat
- **저자**: 어레이도 RMA(이미 정규화) **후 normalizeBetweenArrays로 또 정규화** = 이중 정규화
- **우리**: 어레이는 RMA로 이미 정규화됨 → **추가 정규화 스킵(옵션1)**
- **왜**: 이미 RMA로 정규화된 걸 또 하는 건 중복. (저자는 데이터 합치기 대비/표준관행으로 넣은 듯)
- **차이**: 값 거의 동일, 우리가 더 깔끔(중복 제거). ComBat은 둘 다 sva 표준으로 동일

### 1-3. GSE96804 유전자 수 (같은 데이터, 우리 내부 두 버전)
- **둘 다 GSE96804** (샘플 61명 동일) — 매핑 방식만 다름
    - 기존 재현: 33,720 유전자 / 새 재현(oligo): 23,910 유전자
- **왜 줄었나**: db가 **유전자 이름(심볼)이 붙는 것만** 남김 → 빠진 12,793개는 대부분 **비유전자**
    - 우리가 임의로 지운 게 아니라, **db가 이름을 못 붙여서 자동으로 빠진 것**

**"지워도 되나?" → 됨. 이유 3가지**

1. **표준 절차** — 마이크로어레이 분석에서 "이름 안 붙는 probe 제거"는 정식 방법 (저자도 함)
2. **어차피 못 쓰는 것** — DEG·경로·ML은 전부 **유전자 이름으로** 계산해요.
    이름이 없으면(예: `AC000036.4`) 다음 분석에서 그게 무슨 유전자인지 몰라 **아예 쓸 수가 없음** → 남겨도 무의미
3. **진짜 유전자가 아님** — 빠진 것들은 이런 종류:
    - **안티센스RNA(-AS1)**: 단백질을 안 만드는 RNA
    - **미지영역(AC00..)**: 아직 정체 모르는 유전체 좌표 (이름 대신 번호만)
    - **가성유전자(-P1)**: 기능 잃은 유전자 잔재
    → 셋 다 **단백질을 만드는 진짜 유전자가 아님**. 이 연구가 찾는 건 단백질 유전자(FN1·ALDH2)라 무관.

**"ML 성능 차이 나나?(96804)" → 안 남**
- ML은 전체 유전자가 아니라 **FN1·ALDH2 관련 후보 몇 개만** 학습에 씀
- 그 유전자들은 두 버전 다 살아있음 → **전체 개수가 줄어도 성능 차이 없음**

**"DEG에서 정상/말기 편향 나나?" → 안 남**
- 매핑은 **probe 기준**이라 정상·말기 상관없이 똑같이 적용 → **특정 그룹만 빠지는 일 없음**
- (게다가 DEG는 GSE142025=RNA-seq라 처음부터 심볼이라 매핑 이슈 자체가 없음)

- **결론**: 유전자 수는 줄었지만 **진짜 유전자 손실 없고 결과·성능 영향 없음**. 새 방식이 더 깔끔.
    - ※ 저자 DEG(GSE142025)와는 다른 데이터라 비교 대상 아님

### 1-4. 검증셋 샘플 필터 (104948·104954)
- **저자**: 196개 중 DN(당뇨)·LD(공여자)만 → 33/38개
- **우리**: 동일 (DN→DKD, LD→Control 필터)
- **왜**: 여러 질환 섞인 데이터라 DKD·정상만 선별
- **차이**: 없음 (저자 기준 따름)

### 1-5. 그룹 라벨 (DKD/Control, Early/Late) 붙이는 방법
- **저자**: **s1/s2/s3.txt** — 저자가 GEO 페이지 보고 **손으로 만든 샘플이름 목록** 3개
    (s1=Control, s2=Early, s3=Late) → 코드가 읽어 `GSM_그룹`으로 라벨
- **우리 (최종)**: 로컬 `series_matrix.txt.gz` **메타데이터에서 자동 추출** (diabetic→DKD)
    - (초기엔 `getGEO`(인터넷) 썼으나 오프라인 실패 → series_matrix 메타로 교체, 이제 인터넷 불필요)
- **왜**: s1/s2/s3 파일은 저자가 공유 안 함 + getGEO는 인터넷 필요 → 로컬 메타로 대체
- **차이**: **방법만 다르고 그룹 판정 결과는 동일** (probe→gene 라이브러리와는 무관, 이건 "라벨" 얘기)
    - ✅ 검증: 30529 DKD10/Control12, 96804 DKD41/Control20 — 기존과 그룹 수 일치

## 4. MR 데이터

### 4-1. 노출(exposure) 소스
- **저자/기존**: eqtl-a **VCF 완제품**(beta·SE 있음) → FN1 OR **2.78**
- **우리**: eQTLGen **원본**(Zscore) → 빈도로 beta/SE 변환 → FN1 OR **4.53**
- **왜**: 완제품 말고 원본에서 직접 도구변수를 만들어 검증
- **차이**: **방향·유의성 동일**(위험, p≈0.012). OR 크기만 Z-스케일 차이 → 직접 비교 불가

### 4-2. ALDH2 도구변수 수
- **저자**: 14개 IV → OR 0.673, p=0.032 (유의)
- **우리**: 3개 IV → OR 0.57, p=0.097 (경계)
- **왜**: 엄격 클럼핑(r²<0.001)으로 독립 신호 3개로 축약
- **차이**: **방향 동일**(보호), 유의성만 IV 수(검정력) 차이

### 4-3. GWAS(결과) 데이터 개수/버전
- **저자**: FinnGen **64,663명** / GCST **388,955명** (사람 수, R12, 2025.2.15)
- **우리 파일**: FinnGen **SNP 21,191,346** / GCST **SNP 28,159,587** (행=SNP 수)
- **왜**: ① 저자값=사람수 vs 우리값=SNP수(단위 다름) ② DB 업데이트로 실제 증가 가능
- **차이**: 단위가 달라 직접 비교 불가 + 버전 최신화로 우리가 더 많을 수 있음
    - → 정확한 재현은 **같은 버전(R12)** 사용해야

## 5. 머신러닝

- **저자 코드**: 실제론 **LASSO ∩ SVM-RFE 교집합**
- **우리**: 저자 코드 따름
- **왜**: 본문과 코드가 달라 실제 코드 기준
- **차이**: 세부 절차 다르나 **최종 FN1·ALDH2 동일**

## 한 줄 요약

> 재현의 **방향·결론은 논문과 일치**(FN1=위험, ALDH2=보호). 차이는 주로
> ① **probe매핑 방식**(저자=테이블 다운 / 우리=라이브러리)
> ② **MR 노출 소스**(완제품 VCF vs eQTLGen 원본)
> ③ **도구변수 수·threshold·GWAS 버전**
> 때문이고, 전부 **오류가 아니라 방법·데이터 버전 차이**. 도킹·pseudotime만 재현 제외.
